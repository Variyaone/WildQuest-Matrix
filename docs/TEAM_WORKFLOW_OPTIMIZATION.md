# 团队分工优化方案 v2.0

> 目标：提高团队效率，降低主Agent token消耗，充分利用所有成员

---

## 当前问题分析

### 问题1：小龙虾token消耗过高（75%）
- **原因**：参与太多细节工作
- **后果**：token浪费，响应慢
- **解决**：只做决策，不参与细节

### 问题2：架构师负担过重
- **原因**：所有代码都由架构师写
- **后果**：效率低，质量难保证
- **解决**：分工明确，简单任务分配给创新者

### 问题3：评审官和创新者闲置
- **原因**：工作分配不均
- **后果**：资源浪费，团队效率低
- **解决**：激活闲置成员，承担审核和优化工作

## 新分工方案

### 🦞 小龙虾（main）- 纯决策层

**职责：**
1. 目标解析（解读老大指令，定义终极目标）
2. 任务拆解（将目标拆解为DAG流程）
3. 资源调度（分配任务给专家）
4. 进度监控（监控任务状态，处理异常）
5. 交付验证（验证成果是否符合目标）
6. 决策制定（T0等待用户，非T0立即执行）

**禁止事项：**
- ❌ 写代码
- ❌ 调试程序
- ❌ 执行技术任务
- ❌ 参与细节讨论
- ❌ 频繁回复消息（减少token消耗）

**Token目标：** < 30%（从75%降到30%）

---

### 🏗️ 架构师（architect）- 核心架构层

**职责：**
1. 系统架构设计
2. 核心模块开发
3. 复杂功能实现
4. 技术方案设计
5. 性能优化（核心部分）

**不负责：**
- ❌ 简单脚本开发
- ❌ 文档撰写
- ❌ 测试执行
- ❌ 小功能开发

**Token目标：** ~30%

---

### 🔍 评审官（critic）- 质量管理层

**职责：**
1. 代码审核（审核架构师、创新者的代码）
2. 质量检查（检查代码质量、风格、规范）
3. 风险审查（识别潜在风险）
4. 测试执行（执行功能测试、回归测试）
5. 验证报告（生成验证报告，确认是否符合要求）

**工作流程：**
```
架构师提交代码 → 评审官审核 → 发现问题 → 反馈给架构师
创新者提交代码 → 评审官审核 → 发现问题 → 反馈给创新者
评审官执行测试 → 记录结果 → 生成报告 → 汇报给小龙虾
```

**Token目标：** ~20%

---

### 🚀 创新者（innovator）- 实施优化层

**职责：**
1. 简单脚本开发（小功能、辅助工具）
2. 系统优化（性能优化、效率提升）
3. 新功能探索（实验性功能、原型开发）
4. 文档撰写（技术文档、使用手册）
5. 工具开发（辅助工具、自动化脚本）

**不负责：**
- ❌ 核心架构设计
- ❌ 复杂功能开发
- ❌ 最终决策

**Token目标：** ~20%

---

### ✍️ 创作者（creator）- 内容生产层

**职责：**
1. 报告生成（日报、周报、月报）
2. 文档撰写（非技术文档）
3. 内容整理（知识库维护）
4. 消息推送（非技术推送）

**Token目标：** ~10%


## Token消耗优化策略

### 小龙虾（main）减少token的方法：

1. **减少消息回复**
   - 只在关键节点回复（开始、完成、异常）
   - 不参与细节讨论
   - 用简短指令替代长篇回复

2. **减少文件读取**
   - 只读关键文件（IDENTITY.md、HEARTBEAT.md）
   - 不参与代码细节审查
   - 信任评审官的验证报告

3. **减少命令执行**
   - 只执行关键验证命令
   - 细节执行分配给对应agent
   - 不亲自测试

4. **减少思考输出**
   - 减少自言自语
   - 直接行动，不叙述过程
   - 只在必要时解释决策
