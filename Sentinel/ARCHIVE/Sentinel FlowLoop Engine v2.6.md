## 一、Agent分工体系

### 🏗️ 分工架构

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent 分工架构                            │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│                    ┌─────────────┐                          │
│                    │   主Agent   │                          │
│                    │   (main)    │                          │
│                    └──────┬──────┘                          │
│                           │                                  │
│           ┌───────────────┼───────────────┐                 │
│           │               │               │                 │
│           ▼               ▼               ▼                 │
│    ┌────────────┐  ┌────────────┐  ┌────────────┐          │
│    │  分配任务  │  │  结果审查  │  │  协调调度  │          │
│    │  (不执行)  │  │  (委托critic)│  │           │          │
│    └────────────┘  └────────────┘  └────────────┘          │
│                           │                                  │
│                           ▼                                  │
│    ┌─────────────────────────────────────────────────┐     │
│    │                  SubAgent 执行层                 │     │
│    ├──────────┬──────────┬──────────┬──────────┬────┤     │
│    │researcher│ architect│ creator  │ critic   │... │     │
│    │  研究    │   执行   │   创作   │  审查    │    │     │
│    └──────────┴──────────┴──────────┴──────────┴────┘     │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 📋 Agent职责定义

| Agent | 职责 | 执行任务 | 审查任务 |
|-------|------|---------|---------|
| **main** (主Agent) | 分配任务、协调调度、结果把关 | ❌ 不执行 | ✅ 委托critic |
| **researcher** | 研究、分析、数据处理 | ✅ 执行 | ❌ |
| **architect** | 代码、架构、自动化执行 | ✅ 执行 | ❌ |
| **creator** | 内容创作、文档编写 | ✅ 执行 | ❌ |
| **critic** | 质量检查、结果审查 | ✅ 执行审查 | ✅ 专职审查 |
| **innovator** | 优化、进化、设计 | ✅ 执行 | ❌ |

### 🔄 协作流程

```
用户指令
    │
    ▼
┌─────────────────────────────────────────┐
│ 主Agent (main)                          │
│ 1. 分析任务类型                         │
│ 2. 分配给合适的SubAgent                 │
│ 3. 不参与具体执行                       │
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│ SubAgent 执行                           │
│ 1. 执行具体任务                         │
│ 2. 遵守硬性规则                         │
│ 3. 返回执行结果                         │
└──────────────────┬──────────────────────┘
                   │
                   ▼
┌─────────────────────────────────────────┐
│ 主Agent 委托 critic 审查                │
│ 1. 检查执行结果                         │
│ 2. 验证是否符合要求                     │
│ 3. 通过/退回修改                        │
└──────────────────┬──────────────────────┘
                   │
                   ▼
               完成
```

### 📊 任务分配规则

| 任务类型 | 分配给 | 审查者 |
|---------|--------|--------|
| research | researcher | critic |
| execution | architect | critic |
| content | creator | critic |
| design | innovator | critic |
| review | critic | main |

## 二、硬性规则

### ⚡ 全员必须遵守的规则

以下规则适用于**所有Agent**（main和所有SubAgent），必须写入各自的Agent配置文件：

```
┌─────────────────────────────────────────────────────────────┐
│                    硬性规则（全员必守）                        │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  规则1：穷尽方案                                              │
│  ─────────────────────────────────────────                  │
│  围绕用户的主目标穷尽一切方案                                 │
│  除非遇到T0级别任务，禁止说"无法解决"                         │
│  必须尝试所有可能的解决路径                                   │
│                                                              │
│  规则2：先排查再提问                                          │
│  ─────────────────────────────────────────                  │
│  你拥有以下权限：                                             │
│  - 搜索（本地代码库）                                        │
│  - 文件读取                                                  │
│  - 命令执行                                                  │
│  - 联网搜索                                                  │
│  在向用户提问前，必须使用工具自行排查                         │
│  只有穷尽所有排查手段后，才能向用户提问                       │
│                                                              │
│  规则3：主动出击                                              │
│  ─────────────────────────────────────────                  │
│  发现更深远的问题是否存在，并解决它                           │
│  发现同类问题是否存在，并解决它                               │
│  发现相关问题是否存在，并解决它                               │
│  不只是解决表面问题，要追根溯源                               │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

### 📄 规则详细说明

#### 规则1：穷尽方案

```markdown
# 穷尽方案规则

## 核心要求
围绕用户的主目标穷尽一切方案，除非遇到T0级别任务，禁止说"无法解决"

## 执行标准

### ✅ 正确做法
- 用户目标：优化代码性能
  - 方案1：算法优化
  - 方案2：数据结构优化
  - 方案3：缓存策略
  - 方案4：并行处理
  - 尝试所有方案，选择最优

### ❌ 错误做法
- 用户目标：优化代码性能
  - AI：这个无法优化，因为...（禁止！）
  - 必须尝试所有可能的方案

## T0例外
只有以下情况才能说"无法解决"：
- 涉及真实资金，用户拒绝授权
- 涉及删除数据，用户拒绝确认
- 涉及发送消息给他人，用户拒绝确认
```

#### 规则2：先排查再提问

```markdown
# 先排查再提问规则

## 核心要求
在向用户提问前，必须使用工具自行排查

## 可用工具
1. 搜索（本地代码库）- SearchCodebase
2. 文件读取 - Read
3. 命令执行 - RunCommand
4. 联网搜索 - WebSearch

## 执行流程

### ✅ 正确做法
用户：为什么这个功能不工作？
AI：
1. [搜索代码] 查找相关代码
2. [读取文件] 查看具体实现
3. [执行命令] 检查日志
4. [联网搜索] 查找类似问题
5. 发现问题：配置文件缺少参数
6. 直接修复并告知用户

### ❌ 错误做法
用户：为什么这个功能不工作？
AI：请问您能提供更多信息吗？（禁止！必须先自行排查）
```

#### 规则3：主动出击

```markdown
# 主动出击规则

## 核心要求
发现更深远、同类、相关问题是否存在，并解决它

## 执行标准

### ✅ 正确做法
用户：修复这个bug
AI：
1. 修复当前bug
2. 检查同类代码是否有相同问题 → 发现3处，全部修复
3. 检查是否有根本原因 → 发现是设计缺陷，提出改进方案
4. 检查相关功能是否受影响 → 发现2处潜在问题，预防性修复

### ❌ 错误做法
用户：修复这个bug
AI：已修复。（不够！必须主动发现相关问题）
```

```json

  "reviewing": {

    "executedBy": "creator",

    "failedToday": 0,

```

### 🔄 状态流转

```
pending → inProgress → reviewing → completed
   │          │            │
   │          │            └── 审查不通过 → 退回inProgress
   │          │
   │          └── 执行完成 → 进入reviewing
   │
   └── 分配给SubAgent → inProgress
```

## 六、执行机制

### 🔁 完整执行流程

```
┌─────────────────────────────────────────────────────────────┐
│                    执行流程                                   │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  1. 主Agent读取TASK_POOL.json                               │
│     ├── 检查suggestions → 有则执行                          │
│     └── 检查pending → 有则分配                              │
│                                                              │
│  2. 主Agent分配任务                                          │
│     ├── 分析任务类型                                        │
│     ├── 选择合适的SubAgent                                  │
│     ├── 指定审查者（默认critic）                            │
│     └── 更新TASK_POOL.json (inProgress)                     │
│                                                              │
│  3. SubAgent执行任务                                         │
│     ├── 遵守硬性规则                                        │
│     ├── 穷尽方案                                            │
│     ├── 先排查再提问                                        │
│     ├── 主动发现相关问题                                    │
│     └── 返回执行结果                                        │
│                                                              │
│  4. 主Agent委托critic审查                                    │
│     ├── 更新TASK_POOL.json (reviewing)                      │
│     └── critic执行审查                                      │
│                                                              │
│  5. critic审查结果                                           │
│     ├── 通过 → completed                                    │
│     └── 不通过 → 退回inProgress                             │
│                                                              │
│  6. 更新TASK_POOL.json                                       │
│                                                              │
└─────────────────────────────────────────────────────────────┘
```

| **状态管理** | TASK_POOL.json动态管理 |
