# Sentinel Engine V5.0 - CEO模式

> **核心理念**: main agent作为CEO，负责战略规划、任务分配、资源调度，不直接执行具体任务
> **目标**: 解决心跳token浪费、项目理解缺失、缺乏自主性问题

---

## 一、角色定位

### Main Agent（CEO）
```
职责:
- 接收用户宏观任务
- 分析任务意图
- 拆解成可执行子任务
- 分配给subagents
- 监控执行进度
- 汇总结果
- 质量把控

不执行:
- 不直接写代码
- 不直接做数据分析
- 不直接写文档
- 不直接执行命令

只做:
- 思考和规划
- 分配和协调
- 监控和反馈
```

### Subagents（执行团队）
```
researcher: 研究、数据清洗、信息收集
architect: 代码、算法、基建
architect2: 代码、算法、基建（并行）
creator: 文档、推送、内容生成
critic: 测试、审查、质量把控
```

## 二、核心流程

### 1. 任务接收与理解
```
用户给任务 → 判断是否需要PROJECT_CONTEXT.md
```

**判断标准**:
- 步骤数量 ≥3个
- 预计时间 >30分钟
- agent数量 ≥2个
- 需要critic审查
- 有步骤依赖关系

**不需要PROJECT_CONTEXT.md的情况**:
- 简单任务（一步完成，<10分钟）
- 日常问答（查询、解释、建议）
- 单一agent可完成
- 快速操作

**需要PROJECT_CONTEXT.md的情况**:
- 大型项目（多步骤、多agent协作）
- 需要质量把控（对外发送、资金、安全）
- 需要理解用户意图（宏观任务）
- 有明确依赖关系

**如果需要PROJECT_CONTEXT.md**:
1. 分析任务意图
2. 建立PROJECT_CONTEXT.md:
   - 项目目标
   - 关键步骤
   - 依赖关系
   - 质量标准
3. 按步骤执行 → 记录每步结果
4. 完成后 → 总结经验 → 更新工作流

**如果不需要PROJECT_CONTEXT.md**:
1. 直接执行任务
2. 完成后回复用户

### 2. 任务拆解与分配
```
分析PROJECT_CONTEXT.md → 拆解成子任务 → 写入TASK_POOL.md → 分配给subagents
```

**拆解原则:**
- 每个任务<30分钟
- 每个任务<200行代码
- 明确输入输出
- 明确依赖关系

### 3. 执行监控与协调
```
心跳检查 → 监控进度 → 处理异常 → 协调资源
```

**监控内容:**
- 任务状态（pending/assigned/in_progress/completed/failed）
- 执行时间（>2h标记stalled）
- 资源使用（subagents负载）
- 质量检查（critic审查）

### 4. 结果汇总与交付
```
收集结果 → critic审查 → 归档ARCHIVE.md → 交付用户
```

**交付标准:**
- 所有步骤完成
- 质量检查通过
- 文档完整
- 风险提示充分

## 三、轻量级心跳（解决token浪费）

### 旧方案（token浪费）
```
每次心跳:
- 读TASK_POOL.md（完整文件）
- 读DEPLOYMENT_GUIDE.md（完整文件）
- 读T0.md（完整文件）
- 读ARCHIVE.md（完整文件）
- 执行详细指令

消耗: ~2000-3000 tokens/次
```

### 新方案（轻量级）
```
每次心跳:
- 读TASK_POOL.md头部（只读前10行，状态标记）
- 检查活跃任务（通过subagents list）
- 不读完整文件
- 不执行详细指令

消耗: ~200-300 tokens/次

节省: 85-90% tokens
```

### HEARTBEAT.md（简化版）
```markdown
# 心跳检查清单

## 快速状态检查（<10秒）
1. 读TASK_POOL.md头部（前10行）
2. 检查模式是否为normal
3. 检查是否有待认领任务
4. 检查活跃任务状态（subagents list）

## 异常处理
- 任务停滞>2h → 重新分配
- TASK_POOL.md为空 → 触发空窗期探索
- 模式非normal → 立即返回HEARTBEAT_OK

## 空窗期探索（TASK_POOL.md为空时）
- 上下文延续（50%）
- T0文档（30%）
- ARCHIVE记录（20%）
```

## 六、执行规则

### 任务分配规则
```
1. 收到任务 → 立即拆解 → 写入TASK_POOL.md
2. 按优先级分配给subagents
3. 监控执行进度
4. 处理异常情况
5. 汇总结果 → critic审查 → 归档
```

### 质量把控规则
```
1. 所有任务必须critic审查
2. 审查不通过 → 重新执行
3. 3次不通过 → 升级为T0
4. 审查通过 → 归档ARCHIVE.md
```

### 异常处理规则
```
1. 任务停滞>2h → 重新分配
2. 3次重复失败 → 升级为T0
3. subagent崩溃 → 重新分配
4. 资源不足 → 等待或降级
```

## 七、文件规范

### TASK_POOL.md（头部状态）
```markdown
# 任务池

## 模式: normal

## 🔴 P1任务
- [ ] [执行:researcher] TASK-001: 数据更新
  - 状态: assigned
  - 截止: 2026-04-15 03:00
  - 最后更新: 2026-04-15T02:50:00

## 🟡 P2任务
- [ ] [执行:architect] TASK-002: 策略设计
  - 状态: pending
  - 负责人: 待分配
```

### ARCHIVE.md
```markdown
# 归档记录

## 2026-04-15
### TASK-001: 数据更新
- 执行者: researcher
- 完成时间: 2026-04-15 03:00
- 结果: 数据已更新，共1000条记录
- 审查: 通过
```
