# 工具评估标准作业程序 (SOP)

> 版本: 1.0
> 创建日期: 2026-03-19
> 适用对象: 研究员agent + 评审官agent + 指挥官
> 目的: 系统化评估第三方工具对OpenClaw AI指挥官的适用性

---

## 📋 评估流程概览

```
需求发现 → 快速筛选 → 深度研究 → 对比分析 → 风险评估 → 决策建议 → 实施计划
```

**总耗时**: 约2-4小时（简单工具）至3-7天（复杂系统）

---

## Phase 1: 需求确认（5-10分钟）

### 1.1 识别评估触发条件
- [ ] 用户主动询问："这个工具怎么样？"
- [ ] 研究员发现潜在优化机会
- [ ] 技术债务积累需要替代方案
- [ ] 成本/性能达到瓶颈

### 1.2 明确评估目标
- [ ] **解决的问题**: 当前痛点是什么？
- [ ] **预期收益**: 期望提升什么指标？
- [ ] **约束条件**: 预算、时间、技术栈限制
- [ ] **决策时限**: 需要在何时做出决策？

### 1.3 记录到TRACKER
```markdown
## 待评估工具列表
- [ ] 工具名称 (YYYY-MM-DD) - 状态: 待研究/进行中/已完成
```

---

## Phase 2: 快速筛选（15-20分钟）

### 2.1 基本信息收集
- [ ] **官方来源**: GitHub/GitLab、官网文档
- [ ] **版本状态**: Stable/Beta/Alpha
- [ ] **许可证类型**: MIT/Apache 2.0/GPL/AGPL/Proprietary
- [ ] **活跃度**: 最近提交时间、Issue响应速度
- [ ] **社区规模**: Star数、Fork数、Contributors

**检查清单**:
```
- Star > 100 或有明确实际使用案例
- 最近6个月内有维护活动
- License允许商用且无传染性风险（避免AGPL）
- 有基础文档（README、Quick Start）
```

### 2.2 快速排除条件
**立即拒绝 ❌**:
- License为AGPL/GPLv3（需审查公司政策）
- 已停止维护超过12个月
- 无文档或文档极差
- 与OpenClaw核心架构不兼容

**快速通过 ✅**:
- OpenClaw官方Plugin推荐
- 有明确的OpenClaw集成案例
- 技术栈完全匹配（Node.js/Python/Go等）

### 2.3 输出筛选结果
```markdown
## 工具筛选报告 - [工具名称]

### 基本信息
- 来源: [URL]
- 版本: [版本号]
- License: [许可证类型]
- 活跃度: [Star/Fork/最后更新]

### 快速评估结果
- [ ] 通过筛选 → 进入深度研究
- [ ] 需进一步验证 → 补充信息
- [ ] 拒绝 → 原因: [xxx]
```

## Phase 3: 深度研究（30-60分钟）

### 3.1 功能特性分析
- [ ] **核心功能**: 列出3-5个主要能力
- [ ] **设计场景**: 官方定位的使用场景
- [ ] **技术架构**: 依赖、数据模型、API设计
- [ ] **集成方式**: Plugin、CLI、API、SDK

**检查清单**:
- 功能是否能解决Phase 1中识别的问题？
- 设计场景是否与OpenClaw主agent匹配？
- 技术栈是否与当前环境兼容？

### 3.2 性能指标收集
- [ ] **官方数据**: Benchmark、Performance metrics
- [ ] **社区反馈**: Issue讨论、用户评价
- [ ] **实际案例**: 已使用该工具的项目

**关键指标**:
```
- Token成本变化 (X%)
- 响应速度变化 (X%)
- 任务完成率变化 (X%)
- 资源占用 (CPU/Memory/Disk)
```

## Phase 4: 对比分析（30-60分钟）

### 4.1 建立对比矩阵
- [ ] 识别关键对比维度（5-7项）
- [ ] 多工具横向对比

**对比维度参考**:
```
功能性、性能、集成难度、维护成本、社区支持、License合规、学习曲线
```

### 4.2 收益/成本分析
- [ ] **收益量化**: 转化为可衡量的值（美元/时间/准确率）
- [ ] **成本估算**: 部署时间、迁移成本、维护开销
- [ ] **ROI计算**: 投资回报率预估

## Phase 6: 实施与验证（根据工具复杂度）

### 6.1 实施跟踪
- [ ] 创建实施任务卡片
- [ ] 分配责任人（研究员/架构师/工程师）
- [ ] 设置里程碑和验收标准

### 6.2 验证清单
- [ ] **功能验证**: 所有核心功能正常工作
- [ ] **性能验证**: 达到预期的性能指标
- [ ] **兼容性验证**: 与现有系统无冲突
- [ ] **稳定性验证**: 运行7天无重大bug
- [ ] **成本验证**: 实际成本符合预期

### 6.3 回滚计划
```markdown
## 回滚触发条件
- 出现严重bug影响核心功能
- 性能下降超过X%
- 成本增加超过Y%
- 发现严重安全漏洞

## 回滚步骤
1. 停止新工具服务
2. 恢复前一个稳定状态
3. 验证系统正常
4. 记录问题和根因
5. 修复后重新评估
```

## ⚠️ 常见陷阱与注意事项

### 避免过度乐观
- ✅ 相信官方数据，但要验证
- ✅ 看社区真实反馈，而非营销文案
- ✅ 考虑最坏情况下的性能下降

### 避免技术偏见
- ✅ 不要因为"熟悉"就偏袒某技术栈
- ✅ 要考虑长期维护成本，不只是初期开发
- ✅ 评估工具，而非评估团队

### 避免忽视风险
- ✅ 第三方工具的稳定性风险
- ✅ License合规风险（AGPL/GPLv3）
- ✅ 供应商锁定风险

---

## 📊 评估质量检查

在提交决策报告前，自查以下问题：

- [ ] 是否有足够的数据支持决策？（避免主观判断）
- [ ] 是否考虑了所有相关维度？（5W2H检查）
- [ ] 是否识别并量化了风险？（风险矩阵是否完整）
- [ ] 是否给出了明确的行动建议？（不能只给分析不给结论）
- [ ] 文档是否清晰，第三方能否理解？（可读性检查）
