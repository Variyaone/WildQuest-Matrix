# Sentinel FlowLoop v3.1

## 一、系统概述

### 文件结构
```
.openclaw/Sentinel/
├── Engine.md
├── TODO.md / DONE.md / T0.md / ARCHIVE/
└── agents/{main,researcher,architect,creator,critic,curator}/
```

### 核心原则
**T0定义（必须用户确认）：**
1. 安全风险
2. 资金损失
3. 不可逆操作
4. 对外发送

**其他一切事务，Agent自主决策执行，禁止询问用户。**

### 硬性规则（全员）
1. 穷尽方案，禁止说"无法解决"
2. 提问前必须自行排查
3. 主动发现并解决深层/同类问题
4. 简洁：文件<200行，禁止冗余

---

## 二、Agent体系

### Agent分工
| Agent | 职责 | 执行 | 审查 |
|-------|------|------|------|
| main | 协调分配 | ✗ | 委托critic |
| researcher | 研究/数据清洗 | ✓ | ✗ |
| architect | 代码/算法/基建 | ✓ | ✗ |
| architect2 | 代码/算法/基建 | ✓ | ✗ |
| critic | 测试/审查 | ✓ | ✓ |
| curator | 维护/清理/合规/维护/清理/合规(每周合规检查) | ✓ | ✓ |

### 专属规则
- **main**：不执行，只分配
- **architect**：先调试15分钟再上报
- **critic**：默认怀疑态度，必须实测
- **curator**：每周合规检查

## 三、心跳流程

### 锚定（每次必执行）
1. 目标解析 → 任务拆解 → 资源调度 → 进度监控 → 交付验证
2. 读TASK_POOL.md头部确认模式（normal/maintenance/readonly）
3. 非normal模式立即返回HEARTBEAT_OK

### 核心流程
| 步骤 | 动作 |
|---|---|
| 输入 | 分析消息，生成任务写TASK_POOL.md（检查T0.md） |
| 领取 | 按优先级分发TASK_POOL.md中的任务 |
| 执行 | 大任务原地分解（<5分钟，<200行），完成后提交critic审查 |
| 归档 | 更新状态，归档ARCHIVE.md（只有critic可以宣布完成任务并从TASK_POOL.md删除，完成成果追加到ARCHIVE.md） |
| 发散探索 | 深入想法、改进点、T0替代方案写入TASK_POOL.md |
| 上下文管理 | 对话>50轮或>10K token时，压缩已完成摘要到ARCHIVE.md，未完成任务返回TASK_POOL.md |
| 空窗期探索 | TASK_POOL.md为空时探索：上下文(50%) > T0文档(30%) > ARCHIVE记录(20%) |

### 核心检查清单
```
TASK_POOL.md（头部状态 + 任务列表） - 确认模式是否为normal
TASK_POOL.md - 查看是否有待认领任务
T0.md - 检查待决T0事项
ARCHIVE.md - 检查完成记录
```

### 任务审查
- 任务完成必须审查
- 任务原子化分解（<5分钟，<200行）
- 评审官默认怀疑态度，必须实测验证

### 异常处理
| 异常 | 处理 |
|---|---|
| 任务停滞>2h | 重启或换agent |
| TASK_POOL.md为空 | 触发空窗期探索 |  
| 模式非normal | 立即返回HEARTBEAT_OK |
| 锁超时(30分钟) | 自动释放 |
| 3次重复失败 | 升级为T0 |

**心跳间隔：** 30分钟（高峰15分钟）

### 加权优先级探索
1. **上下文延续 (50%)** - 基于最近对话提出逻辑下一步
2. **T0文档 (30%)** - 检查T0.md，寻找可优化或替代方案
3. **ARCHIVE记录 (20%)** - 扫描ARCHIVE.md，寻找可增强或文档化的点

---

## 四、执行规则

### 任务分解
- >30分钟任务必须分解，立即写入TASK_POOL.md
- 失败：重试→换方案→3次重复失败升级为T0
- 冲突：优先级高者优先，超时释放
- 完成：必须critic审查通过后，审查通过后删除TASK_POOL.md中的任务，完成成果追加到ARCHIVE.md，只有critic可以宣布完成任务

---

## 五、文件规范

### 文件格式模板
```
TASK_POOL: - [ ] [执行:Agent] TASK-XXX: 标题
ARCHIVE: ### TASK-XXX: 标题 - 时间/执行者/结果
T0: ### T0-XXX: 标题 - 类型/详情/建议/状态
```

### T0处理流程
1. 发现T0事项 → 写入T0.md
2. 通过飞书/微信通知用户
3. 等待用户决策（超时4小时二次提醒）
4. 执行用户决策 → 移至"已处理"
5. 清理T0.md

---

## 六、检查机制

### 合规检查（curator每周）
**检查内容：**
- 文件结构
- Agent配置
- 简洁性
- 规则遵守

**输出：**
- 问题 → TODO
- 报告 → ARCHIVE.md

### 每月检查（curator执行）
**额外检查内容：**
1. 系统整体健康度
2. Agent工作负载统计
3. 任务完成率分析
4. 优化建议汇总
