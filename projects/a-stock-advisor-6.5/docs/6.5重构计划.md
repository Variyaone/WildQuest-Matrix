### 层级职责与依赖关系

| 层级 | 职责 | 依赖上层 | 被下层依赖 |
|------|------|----------|------------|
| **支撑层** | 调度、监控、存储、日志、配置 | 无（独立） | 所有层 |
| **数据层** | 数据获取、清洗、存储 | 支撑层 | 因子层 |
| **因子层** | 因子计算、验证、挖掘 | 数据层、支撑层 | 信号层 |
| **信号层** | 信号生成、过滤、评估 | 因子层、支撑层 | 策略层 |
| **策略层** | 策略设计、选股、优化 | 信号层、支撑层 | 组合优化层 |
| **组合优化层** | 权重优化、中性化、约束 | 策略层、支撑层 | 风控层 |
| **风控层** | 事前/事中/事后风控 | 组合优化层、支撑层 | 执行层 |
| **执行层** | 实盘交易/仿真回测 | 风控层、支撑层 | 分析层 |
| **分析层** | 绩效分析、归因、报告 | 执行层、支撑层 | 用户 |

### 数据流与输出契约

| 阶段 | 输入 | 输出 | 输出格式 | 传递给 | 数据契约 |
|------|------|------|----------|--------|----------|
| 支撑层 | 配置文件 | 调度任务/存储服务 | - | 所有层 | 统一接口 |
| 数据层 | 外部API | 市场数据 | DataFrame[OHLCV] | 因子层 | H1-H5验证 |
| 因子层 | 市场数据 | 因子值 | DataFrame[stock, factor] | 信号层 | IC/IR验证 |
| 信号层 | 因子值 | 信号强度 | Dict[stock, score] | 策略层 | 胜率验证 |
| 策略层 | 信号强度 | 选股结果 | Dict[stock, selection_score] | 组合优化层 | 回测验证 |
| 组合优化层 | 选股结果 | 目标持仓 | Dict[stock, target_weight] | 风控层 | 权重归一化 |
| 风控层 | 目标持仓 | 审批结果 | List[trade_order] | 执行层 | 风控规则 |
| 执行层(实盘) | 审批结果 | 成交回报 | Dict[trade_result] | 分析层 | 成交确认 |
| 执行层(仿真) | 审批结果 | 回测报告 | Dict[backtest_result] | 分析层 | 撮合记录 |
| 分析层 | 交易结果 | 分析报告 | HTML/PDF | 用户 | 指标完整 |

### 统一数据传递规范

```
数据传递原则:
1. 每层只依赖直接上层，不跨层调用
2. 所有输出必须符合数据契约
3. 支撑层提供统一的数据存取接口
4. 异常情况通过支撑层统一告警

数据格式规范:
├── DataFrame: 用于批量数据（因子值、行情数据）
├── Dict: 用于映射关系（股票→权重、股票→信号）
├── List: 用于列表数据（交易指令、持仓列表）
└── JSON: 用于配置和元数据

存储路径规范:
├── data/master/     # 主数据（持久化）
├── data/cache/      # 缓存数据（临时）
├── data/metadata/   # 元数据索引
├── data/factors/    # 因子数据
├── data/signals/    # 信号数据
├── data/portfolio/  # 组合数据
└── data/reports/    # 报告数据
```

### 因子库、信号库、策略库关系

```
因子库 (原材料)          信号库 (半成品)           策略库 (成品)
┌─────────────┐         ┌─────────────┐         ┌─────────────┐
│ F00001 动量 │ ──────▶ │ S00001 多因子│ ──────▶ │ ST00001 量化│
│ F00002 价值 │  组合    │     选股信号 │  配置    │     选股策略│
│ F00003 质量 │         │             │         │             │
│ F00004 波动│         │ S00002 反转 │         │ ST00002 择时│
│ ...        │         │     信号    │         │     策略    │
└─────────────┘         └─────────────┘         └─────────────┘

调用链路:
策略配置 → 信号配置 → 因子配置 → 计算因子值 → 生成信号 → 选股
```

### 每日执行管线

```
每日/每周执行流程:

1. 数据更新 (Step1)
   └─ 增量更新市场数据 → 更新元数据索引

2. 因子计算 (Step2)
   └─ 遍历因子库 → 计算所有因子值 → 存入因子缓存

3. 信号生成 (Step3a)
   └─ 遍历信号库 → 应用因子组合规则 → 生成信号强度

4. 策略执行 (Step3b)
   └─ 选择活跃策略 → 应用信号 → 生成选股结果

5. 组合优化 (Step4)
   └─ 应用优化方法 → 计算目标权重 → 中性化处理

6. 风控检查 (Step5)
   └─ 事前风控 → 通过则生成交易指令

7. 交易执行 (Step6)
   └─ 订单路由 → 执行算法 → 发送券商

8. 监控报告 (Step7)
   └─ 实时监控 → 绩效归因 → 风险报告
```

```
def migrate_market_data():
    """迁移市场数据"""
    old_path = "data/stocks/"
    new_path = "data/master/stocks/daily/"
    
    # 字段映射
    field_mapping = {
        "日期": "date",
        "开盘": "open",
        "收盘": "close",
        "最高": "high",
        "最低": "low",
        "成交量": "volume",
        "股票代码": "stock_code"
    }
    
    for file in glob(f"{old_path}/*.parquet"):
        df = pd.read_parquet(file)
        df = df.rename(columns=field_mapping)
        df = normalize_stock_code(df)
        df.to_parquet(f"{new_path}/{file.name}")
```

```
def retry_with_backoff(func, max_retries=5):
    for i in range(max_retries):
        try:
            return func()
        except NetworkError as e:
            if i == max_retries - 1:
                raise
            wait_time = (2 ** i) + random.random()
            time.sleep(wait_time)
```

**核心功能与流程**：

**1. 前置检查管理器 (PreCheckManager)**
- **功能**：统一管理所有模块的前置检查
- **核心逻辑**：
  - 定义各层的硬性要求（H）、弹性要求（E）、边际要求（M）
  - 按顺序执行检查
  - 记录检查结果
  - 管理结果可信度
- **检查规则**：
  ```
  检查规则：
  
  🔴 硬性要求（H）：必须满足，否则一票否决
  - 失败后果：流程终止，不可进入下一步
  - 处理方式：返回失败步骤，修复后重新检查
  - 结果标记：标记为"不可信"，需从头修复
  
  🟡 弹性要求（E）：有条件可适当让步
  - 失败后果：记录偏差，评估风险后可继续
  - 处理方式：记录警告，标记风险等级
  - 结果标记：标记为"部分可信"，记录偏差
  
  🟢 边际要求（M）：锦上添花，没有不影响
  - 失败后果：不影响流程，记录为优化项
  - 处理方式：记录建议，不阻塞流程
  - 结果标记：不影响可信度
  ```

**2. 结果可信度管理 (TrustManager)**
- **功能**：管理整个管线的输出结果可信度
- **核心逻辑**：
  ```
  可信度状态：
  ├── TRUSTED：所有硬性要求通过，可信
  ├── PARTIAL：弹性要求有偏差，部分可信
  └── UNTRUSTED：硬性要求失败，不可信
  
  可信度传递规则：
  1. 任一步骤硬性要求失败 → 整体结果标记为UNTRUSTED
  2. 任一步骤弹性要求偏差 → 整体结果标记为PARTIAL（如果无硬性失败）
  3. 所有检查通过 → 整体结果标记为TRUSTED
  
  失败回退机制：
  1. 检测到UNTRUSTED → 返回管线最前端（Step1数据层）
  2. 从失败步骤开始，按顺序重新执行
  3. 每个步骤执行后立即检查
  4. 直到所有步骤通过或用户中断
  
  用户中断处理：
  - 用户可随时中断修复流程
  - 中断时记录当前状态
  - 下次可从断点继续
  ```

```
├── 深圳交易所: .SZ (如: 000001.SZ, 300001.SZ)
├── 上海交易所: .SH (如: 600000.SH, 688001.SH)
└── 北京交易所: .BJ (如: 430001.BJ)

各数据源转换规则:
├── AKShare: 已是标准格式，无需转换
├── BaoStock: sh.600000 → 600000.SH, sz.000001 → 000001.SZ
├── yfinance: 600000.SS → 600000.SH, 000001.SZ → 000001.SZ
└── Ashare: 已是标准格式，无需转换

转换函数:
def normalize_stock_code(code: str, source: str) -> str:
    if source == 'baostock':
        # sh.600000 → 600000.SH
        parts = code.split('.')
        return f"{parts[1]}.{parts[0].upper()}"
    elif source == 'yfinance':
        # 600000.SS → 600000.SH
        return code.replace('.SS', '.SH')
    else:
        return code
```

- [ ] 15.9 运行测试确保风控功能正常

**核心功能与流程**：

**1. 事前风控 (PreTradeRisk)**
- **功能**：交易前检查，防止违规交易
- **核心逻辑**：
  - 单票权重上限检查（≤12%）
  - 行业集中度检查（≤30%）
  - 总仓位上限检查（≤95%）
  - 止损线检查（回撤≤15%）
  - 黑名单检查
- **主要流程**：
  ```
  事前检查：
  接收交易指令 → 逐项检查风控规则 → 记录违规项
  → 通过则放行/拒绝则拦截 → 返回检查结果
  ```

**2. 事中风控 (IntradayRisk)**
- **功能**：实时监控交易和持仓风险
- **核心逻辑**：
  - 实时盈亏监控
  - 持仓风险监控
  - 异常交易检测
  - 紧急止损触发
- **主要流程**：
  ```
  实时监控：
  获取实时行情 → 计算实时风险指标 → 检查阈值
  → 超阈值则预警/止损 → 记录监控日志
  ```

**3. 事后风控 (PostTradeRisk)**
- **功能**：交易后分析和归因
- **核心逻辑**：
  - 绩效归因分析
  - 风险分解分析
  - 回撤分析
  - 合规检查
- **主要流程**：
  ```
  事后分析：
  获取交易记录 → 计算风险指标 → 归因分析
  → 生成风险报告 → 提交审计
  ```

**4. 风险指标计算 (RiskMetrics)**
- **功能**：计算各类风险指标
- **核心指标**：
  - VaR（风险价值）
  - CVaR（条件风险价值）
  - 最大回撤
  - 波动率
  - Beta/相关性
- **主要流程**：
  ```
  计算流程：
  获取持仓和收益数据 → 应用计算模型 → 输出风险指标
  ```

**5. 风险预警 (RiskAlert)**
- **功能**：风险超阈值时发送预警
- **预警类型**：
  - 回撤预警（回撤>10%）
  - 集中度预警（行业>25%）
  - 流动性预警（成交量异常）
  - 波动率预警（波动率飙升）
- **主要流程**：
  ```
  预警流程：
  监控风险指标 → 超阈值触发 → 生成预警消息
  → 发送通知（邮件/短信/钉钉） → 记录预警日志
  ```

**硬性要求验证（H1-H5）**：
| 编号 | 要求 | 检查内容 | 失败后果 |
|------|------|----------|----------|
| H1 | 单票权重上限 | ≤12% | 个股风险过大 |
| H2 | 行业集中度 | ≤30% | 行业风险过大 |
| H3 | 总仓位上限 | ≤95% | 无缓冲资金 |
| H4 | 止损线检查 | 回撤≤15% | 本金损失过大 |
| H5 | 黑名单检查 | 无禁买股票 | 合规风险 |

| 编号 | 要求 | 理想标准 | 让步条件 |

| E3 | 流动性 | 成交额>1000万 | 小盘策略 |
| E4 | 相关性 | 股票间<0.7 | 行业集中策略 |

**核心功能与流程**：

**1. 订单管理 (OrderManager)**
- **功能**：管理交易订单生命周期
- **核心逻辑**：
  - 订单生成：根据目标持仓计算买卖数量
  - 订单状态管理：待执行/执行中/已成交/已取消
  - 订单查询和统计
- **主要流程**：
  ```
  订单流程（手动交易模式）：
  接收目标持仓 → 计算差额 → 生成订单
  → 分配订单ID → 推送给交易员 → 等待交易员确认
  → 接收成交录入 → 更新订单状态
  ```

**2. 持仓管理 (PositionManager)**
- **功能**：管理当前持仓
- **核心逻辑**：
  - 持仓记录：记录每只股票的持仓数量和成本
  - 持仓更新：根据手动录入的成交更新持仓
  - 持仓查询：支持多维度查询
- **主要流程**：
  ```
  持仓更新：
  接收手动录入的成交 → 更新持仓数量 → 更新持仓成本
  → 计算持仓盈亏 → 保存持仓记录
  ```

**3. 交易报告生成器 (TradeReportGenerator)**
- **功能**：生成详细的交易决策报告，供交易员审核和手动下单
- **核心逻辑**：
  - 汇总交易决策依据
  - 生成可读性报告
  - 支持多种格式输出
- **报告内容**：
  ```
  交易报告结构：
  ├── 基本信息
  │   ├── 报告日期/时间
  │   ├── 策略名称/编号
  │   └── 账户信息
  ├── 交易指令【核心】
  │   ├── 目标持仓列表
  │   ├── 需要买入的股票
  │   │   ├── 股票代码/名称
  │   │   ├── 建议买入数量
  │   │   ├── 建议买入价格区间
  │   │   ├── 预计买入金额
  │   │   └── 建议执行时机
  │   ├── 需要卖出的股票
  │   │   ├── 股票代码/名称
  │   │   ├── 建议卖出数量
  │   │   ├── 建议卖出价格区间
  │   │   ├── 预计卖出金额
  │   │   └── 建议执行时机
  │   └── 预计总交易金额
  ├── 决策依据
  │   ├── 因子信号（各因子值及权重）
  │   ├── 信号强度评分
  │   ├── 组合优化方法
  │   └── 风控检查结果
  ├── 风险评估
  │   ├── 单票权重分布
  │   ├── 行业集中度
  │   ├── 预计换手率
  │   └── 风险指标（VaR等）
  ├── 历史表现参考
  │   ├── 策略历史胜率
  │   ├── 近期收益曲线
  │   └── 最大回撤
  └── 附录
      ├── 相关股票基本面摘要
      ├── 近期新闻/公告
      └── 技术指标图表
  ```
- **主要流程**：
  ```
  报告生成：
  收集交易决策 → 汇总决策依据 → 计算风险指标
  → 生成报告文档 → 推送给交易员
  ```

**4. 交易报告推送 (TradeNotifier)**
- **功能**：推送交易指令给交易员，等待确认
- **推送时机**：
  - 交易前：推送交易决策报告
  - 交易中：接收交易员反馈
  - 交易后：推送成交确认
- **推送渠道**：
  - 邮件
  - 钉钉/企业微信
  - 短信（紧急情况）
- **主要流程**：
  ```
  推送流程：
  生成交易报告 → 选择推送渠道 → 发送通知
  → 等待交易员确认 → 记录推送日志
  ```

**5. 手动成交录入 (ManualEntry)**
- **功能**：交易员手动录入实际成交情况
- **核心逻辑**：
  - 接收交易员录入的成交信息
  - 验证成交数据有效性
  - 更新订单状态
  - 更新持仓记录
- **录入内容**：
  ```
  成交录入表单：
  ├── 订单ID（关联原订单）
  ├── 股票代码
  ├── 买卖方向
  ├── 成交数量
  ├── 成交价格
  ├── 成交时间
  └── 备注（可选）
  ```
- **主要流程**：
  ```
  录入流程：
  接收成交数据 → 验证数据有效性 → 更新订单状态
  → 更新持仓记录 → 生成成交确认 → 推送确认通知
  ```

**6. 交易员确认机制 (TraderConfirmation)**
- **功能**：确保交易员收到并确认交易指令
- **核心逻辑**：
  - 发送交易指令后等待确认
  - 超时未确认自动提醒
  - 记录确认时间和确认人
- **确认类型**：
  ```
  确认类型：
  ├── 已收到：交易员确认收到报告
  ├── 已执行：交易员确认已下单
  ├── 部分执行：交易员确认部分成交
  ├── 暂停执行：交易员决定暂停
  └── 拒绝执行：交易员决定不执行
  ```
- **主要流程**：
  ```
  确认流程：
  推送交易指令 → 等待确认（超时30分钟）
  → 未确认则提醒 → 记录确认结果
  ```

**7. 交易状态跟踪 (TradeStatusTracker)**
- **功能**：跟踪交易执行状态
- **核心逻辑**：
  - 跟踪每笔订单的执行状态
  - 统计执行完成率
  - 记录未执行原因
- **状态类型**：
  ```
  订单状态：
  ├── 待推送：订单已生成，等待推送
  ├── 已推送：订单已推送给交易员
  ├── 已确认：交易员已确认收到
  ├── 执行中：交易员正在执行
  ├── 已成交：交易已完成
  ├── 部分成交：部分完成
  ├── 已取消：订单已取消
  └── 已拒绝：交易员拒绝执行
  ```
- **主要流程**：
  ```
  状态跟踪：
  推送订单 → 记录推送时间 → 等待确认
  → 更新状态 → 统计执行情况
  ```

**硬性要求验证（H1-H4）**：
| 编号 | 要求 | 检查内容 | 失败后果 |
|------|------|----------|----------|
| H1 | 订单有效性 | 数量>0, 价格合理 | 交易失败 |
| H2 | 资金充足 | 可用资金≥所需资金 | 交易失败 |
| H3 | 持仓充足 | 卖出数量≤持仓数量 | 交易失败 |
| H4 | 交易时间 | 在交易时段内 | 交易失败 |

**核心功能与流程**：

**1. 每日任务调度器 (DailyScheduler)**
- **功能**：管理每日收盘后的任务执行顺序
- **任务顺序**：
  ```
  每日收盘后执行流程：
  
  1. 数据更新（15:30-16:00）
     └─ 增量更新市场数据 → 更新元数据索引
  
  2. 数据过期检查（16:00-16:15）
     └─ 检查因子缓存是否过期 → 检查信号数据是否过期
     └─ 判断哪些需要重新计算
  
  3. 因子计算（16:15-17:00）【仅计算过期因子】
     └─ 遍历需要更新的因子 → 计算因子值 → 存入因子缓存
  
  4. 信号生成（17:00-17:30）【仅生成过期信号】
     └─ 遍历需要更新的信号 → 应用因子组合规则 → 生成信号强度
  
  5. 策略执行（17:30-18:00）
     └─ 选择活跃策略 → 应用信号 → 生成选股结果
  
  6. 组合优化（18:00-18:30）
     └─ 应用优化方法 → 计算目标权重 → 中性化处理
  
  7. 风控检查（18:30-19:00）
     └─ 事前风控 → 生成交易指令
  
  8. 日报生成（19:00-19:30）
     └─ 汇总今日数据 → 生成绩效报告 → 推送给相关人员
  
  9. 数据备份（19:30-20:00）
     └─ 备份今日数据 → 清理临时文件
  
  注意：
  - 因子计算和信号生成只处理过期/缺失的数据
  - 周线因子只在周一计算
  - 月线因子只在月初计算
  - 财务因子只在财报发布后计算
  ```

**2. 数据更新器 (DailyDataUpdater)**
- **功能**：每日收盘后增量更新数据
- **核心逻辑**：
  - 检测数据缺口
  - 增量下载缺失数据
  - 更新元数据索引
- **主要流程**：
  ```
  数据更新：
  获取交易日历 → 检测缺失日期 → 增量下载
  → 数据清洗 → 更新主数据 → 更新元数据
  ```

**3. 因子计算器 (DailyFactorCalculator)**
- **功能**：智能判断是否需要计算因子值
- **核心逻辑**：
  - 检查因子数据是否过期
  - 根据因子类型决定计算频率
  - 只计算需要更新的因子
- **因子计算频率建议**：
  ```
  因子类型与计算频率：
  
  | 因子类型 | 计算频率 | 原因 |
  |----------|----------|------|
  | 日线动量因子 | 每日收盘后 | 依赖最新收盘价 |
  | 日线成交量因子 | 每日收盘后 | 依赖最新成交量 |
  | 周线因子 | 每周一收盘后 | 依赖周线数据 |
  | 月线因子 | 每月首个交易日收盘后 | 依赖月线数据 |
  | 财务因子 | 财报发布后 | 依赖财报数据 |
  | 技术指标因子 | 每日收盘后 | 依赖最新行情 |
  | 基本面因子 | 每周/每月 | 变化较慢 |
  
  智能计算策略：
  1. 检查因子缓存是否存在
  2. 检查因子数据时间戳
  3. 检查依赖的数据是否更新
  4. 判断是否需要重新计算
  ```
- **数据过期检查**：
  ```
  过期检查逻辑：
  
  1. 检查因子缓存时间戳
     - 如果缓存时间 < 最新数据时间 → 需要重算
     
  2. 检查依赖数据是否更新
     - 如果依赖的市场数据更新 → 需要重算
     - 如果依赖的财务数据更新 → 需要重算
     
  3. 检查因子类型
     - 日线因子：每日重算
     - 周线因子：每周重算
     - 月线因子：每月重算
     
  4. 返回需要重算的因子列表
  ```
- **主要流程**：
  ```
  因子计算：
  加载市场数据 → 检查因子缓存 → 判断是否需要重算
  → 只计算过期因子 → 验证因子值 → 存入因子存储
  ```

**4. 信号生成器 (DailySignalGenerator)**

- **核心逻辑**：
  - 检查信号数据是否过期

  ```
  信号类型与生成频率：
  
  | 信号类型 | 生成频率 | 原因 |

  | 周线选股信号 | 每周一收盘后 | 依赖周线因子 |
  | 月线选股信号 | 每月首个交易日收盘后 | 依赖月线因子 |

  | 行业轮动信号 | 每日收盘后 | 依赖行业数据 |
  
  智能生成策略：

  2. 检查依赖的因子是否更新
  ```

- **主要流程**：

- **衰减处理建议**：
  ```
  衰减处理策略：
  - 轻度衰减：降低因子权重
  - 中度衰减：暂停使用因子
  - 重度衰减：从因子库移除
  - 过度拥挤：降低因子权重
  ```
- **主要流程**：
  ```
  衰减监控：
  每日计算因子指标 → 对比历史均值 → 判断衰减程度
  ```
