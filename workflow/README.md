## 🎯 核心目标

Sentinel 是 AI Agent 团队的自主工作流引擎，实现任务自动化、上下文压缩、模式学习和资源调度。

### 三大支柱
1. **任务池机制** - 自动化任务分配与状态跟踪
2. **上下文压缩** - 智能管理对话长度，避免 token 浪费
3. **模式学习** - 从历史中学习，主动优化

```
    └── PATTERNS.md         # 模式学习库
```

## 🔁 10步心跳工作流

1. **自检与恢复** - 检查核心文件、存储空间、JSON 格式
2. **T0任务监控** - 扫描阻塞任务，处理超时
3. **输入处理与任务化** - 分析新对话，生成新任务
4. **上下文压缩与摘要** - 检测过长对话，自动压缩
5. **任务领取与锁** - 按优先级领取任务，负载均衡检查
6. **任务执行与状态更新** - 实时更新进度，原地分解子任务
7. **任务完成与归档** - 完成归档，释放锁，依赖检查
8. **主动探索** - 空窗期触发（40%延续 + 35%复盘 + 25%全局）
9. **维护与清理** - 清理超过7天的完成任务
10. **用户通知决策** - 按通知矩阵（T0立即，P0可选）

## 🎭 Agent 能力注册

每个 SubAgent 必须在 `agents/REGISTRY.md` 中注册：

- **最大并发任务数** - 防止过载
- **专长领域** - 任务分配依据
- **能力声明** - 具体能力列表
- **最后更新时间** - 确保信息最新

## 🔬 任务分解原则（T0级规则）

### 原子化最小单元
**所有任务必须分解为可独立执行的原子单元。**

❌ **错误示例**（任务过大）：
```
实现一个完整的因子挖掘模块
```

✅ **正确示例**（原子化任务）：
```
TASK-001a: 创建factor/base.py基类
TASK-001b: 实现momentum_factor函数
TASK-001c: 编写test_momentum_factor.py
TASK-001d: 运行 pytest tests/factor/ -v
TASK-001e: git提交factor模块
```

### 分解标准
- **单次执行 < 5分钟** → 合理大小
- **代码量 < 200行** → 避免上下文过长
- **独立测试覆盖** → 每个任务有可验证的输出
- **可独立git提交** → 每个单元完成即可提交

### 上下文管理
- 每任务携带最小依赖信息
- 避免传递完整项目代码
- 用文件路径引用而非文件内容
- 动态读取配置文件而非硬编码

## 🔒 任务锁机制

防止任务冲突和重复执行：

- **超时释放**：锁超时（30分钟）自动释放
- **负载检查**：领取前检查 Agent 当前负载
- **状态同步**：LOCK_STATE.json 实时更新

---

## 🧠 决策分级

| 级别 | 类型 | 处理方式 |
|------|------|----------|
| T0 | 真实资金、致命故障、不可逆操作 | 等待用户确认 |
| P0 | 安全风险调查 | 评估后可能转为T0 |
| P1 | 高优先级任务 | 立即执行，事后汇报 |
| P2 | 中优先级任务 | 正常处理 |
| P3 | 低优先级任务 | 有空时处理 |

---

## 🚨 智能升级机制

### 优先级衰减
普通任务超过预期耗时2倍未开始 → 优先级自动提升一级

### 失败升级
普通任务失败3次后：
1. 分析失败模式 → 写入 PATTERNS.md
2. 创建 P0 调查任务
3. 评估是否触及安全边界
4. 决定创建 T0 任务或调整策略重试

---

## 💡 主动探索（空窗期驱动）

触发条件：TASK_POOL.md 中无待认领的 P1 以上任务

## 加权优先级探索

### 1. 上下文延续 (40%)
- 基于 CONTEXT_SUMMARY.md
- 提出逻辑下一步
- 保持任务连贯性

### 2. 模式复盘 (35%)
- 分析 PATTERNS.md 中的失败或高频模式
- 生成优化任务
- 持续改进系统

### 3. 全局改进 (25%)
- 扫描 ARCHIVE.md
- 寻找可增强、文档化或自动化的点
- 提升整体效率

## ✅ 任务完成标准（T0级规则）

### git提交是任务完成的唯一证明
**任何声称"完成"的任务必须满足：**

1. **测试通过** → pytest显示100% passed
2. **代码可运行** → 实际执行无报错
3. **文件保存完整** → ls -la确认所有输出文件存在
4. **git已提交** → 看到commit hash（唯一标志）

❌ **不接受的"完成"**：
- "代码写完了"（未测试）
- "应该能跑"（未实际运行）
- "功能实现了"（未验证）
- "我检查过了"（无命令输出）

✅ **接受的"完成"**：
```
✅ pytest passed: 20/20 (3.2s)
✅ 代码可运行: python main.py --test 输出正常
✅ 文件已保存: data/factor_pool.pkl (2.3MB)
✅ 已提交git: abc123def - feat: 新增长期收益因子
任务完成！
```

### 指挥官验证职责
当subagent声称"完成"时：
1. 要求完整的验证命令输出（不省略）
2. 检查是否包含git commit hash
3. 如缺少任何一项 → 打回重做
4. 验证通过 → 归档任务，释放锁

## ⚠️ 必须遵守

### ✅ 强制规则
- 读取 SENTINEL_LOG.json 确认系统模式，非normal模式立即返回
- 任务领取前必须检查 LOCK_STATE.json
- 任务完成后必须释放锁并归档到 ARCHIVE.md
- 任务失败3次必须分析模式并创建P0调查任务
- **空窗期必须触发主动探索**（按40%/35%/25%权重）
- 长对话必须压缩摘要到 CONTEXT_SUMMARY.md

### ❌ 禁止行为
- ❌ 在 readonly 模式下执行任务
- ❌ 跳过锁检查直接领取任务
- ❌ 任务完成后不归档
- ❌ 失败后不分析模式直接重试
- ❌ 空窗期不主动探索
